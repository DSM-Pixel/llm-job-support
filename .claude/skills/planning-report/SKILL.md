# /planning-report

기획 문서 초안을 만든다. 빈 종이에서 시작하지 말고 양식을 채워준다.

## 입력 수집

`AskUserQuestion`으로:
1. **문서 종류**: 기획 보고서 / 기능 정의서 / 아키텍처 설계안 / 발표자료
2. **팀명/세부 과제** (모르면 `teams/` 폴더 스캔하여 선택지로 보여준다)
3. **시연 시나리오 한 줄** (예: "포트홀 영역을 빨갛게 표시해줘")

## 생성 위치

```
docs/
├── <팀명>-<문서종류>.md          # 예: team-1-기획보고서.md
└── slides/<팀명>.md              # 발표자료는 slides/ 아래
```

이미 같은 경로가 있으면 **덮어쓰지 말고 `-v2.md` suffix**를 붙인다.

## 문서별 템플릿

## 4. 기능 정의

### 2) 기능 정의서

표 형식 위주. 한 기능당 행 하나. 각 기능에 **시연 시나리오 1줄** 필수.

### 3) 아키텍처 설계안

- 컴포넌트 다이어그램(ASCII) + 컴포넌트별 한 단락 설명
- 데이터 흐름(요청 → 응답까지)
- 외부 의존성(공공데이터 API, 모델 weight, DB)

### 4) 발표자료 (Marp 호환)

```markdown
---
marp: true
theme: default
paginate: true
---

# <팀명>: <세부 과제명>

8주, 4명, 시연 가능한 프로토타입

---

## 문제

<현장에서 마주한 문제 1-2줄 + 사진/예시>

---

## 우리 접근

- 기술 1: …
- 기술 2: …
- 기술 3: …

---

## 시연

> "<시연 시나리오>"

(영상/스크린샷)

---

## 결과 / 한계 / 다음 단계
```

## 작성 가이드

- 모든 섹션을 비워두지 말 것. 모르면 `<TODO: …>` 마커로라도 표시.
- 표는 가능하면 5행 이하 (가독성).
- 마크다운 표는 컬럼 너비 맞추지 않아도 됨 (어차피 렌더링).
- 발표자료 슬라이드당 **5줄 이내**.

## 마지막 행동

1. 생성한 파일 경로 알림.
2. `planning-writer` 에이전트를 부르면 본문을 채워주겠다고 안내.
3. 빠진 정보(데이터셋 이름, 모델 후보 등)는 체크리스트로 정리.
